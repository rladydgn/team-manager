# Team Manager 프론트엔드 헌법

이 문서는 `fe` 프로젝트의 SDD 기준이다. 모든 화면, 상태 관리, API
연동, 검증 작업은 이 원칙을 따른다.

## 핵심 원칙

### 1. 명세가 먼저다

모든 프론트엔드 기능은 구현 전에 명세로 시작한다. 명세에는 사용자
목표, 화면 범위, 상태, 인수 기준, 제외 범위가 포함되어야 한다.

### 2. 사용자 흐름을 완성한다

화면 단위가 아니라 사용자가 끝내려는 일을 기준으로 설계한다. 로딩,
빈 상태, 실패 상태, 권한 부족 상태, 성공 상태를 함께 정의한다.

### 3. API 계약을 추측하지 않는다

백엔드의 요청/응답/에러 형식은 문서화된 계약을 기준으로 사용한다.
계약이 없거나 모호하면 구현 전에 명세에 질문으로 남긴다.

### 4. 접근성과 반응형을 기본으로 한다

주요 흐름은 키보드와 스크린 리더 사용성을 고려한다. 모바일과 데스크톱
뷰포트에서 텍스트와 컨트롤이 겹치지 않아야 한다.

### 5. 검증 가능해야 완료다

인수 기준은 테스트, lint, 빌드, 또는 수동 검증 절차로 확인 가능해야
한다. UI 변경은 핵심 화면 상태를 검증한다.

### 6. 작고 일관된 구현을 선택한다

현재 코드베이스의 App Router, feature/shared 구조, API helper 패턴을
우선한다. 새 상태 관리 도구나 UI 라이브러리는 명확한 이점이 있을 때만
추가한다.

## 품질 게이트

- 명세에 구현을 막는 모호함이 없어야 한다.
- 사용자 흐름의 성공/실패/빈 상태가 정의되어야 한다.
- API 계약과 에러 처리가 문서화되어야 한다.
- 접근성과 반응형 검증 방법이 있어야 한다.
- `npm run lint`와 필요한 빌드/수동 검증을 통과해야 한다.
