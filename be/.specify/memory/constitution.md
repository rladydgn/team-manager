# Team Manager 백엔드 헌법

## 핵심 원칙

### I. 명세는 단일 진실 공급원이다

모든 백엔드 기능은 구현을 시작하기 전에 반드시 작성된 명세에서
출발해야 한다. 명세는 사용자 또는 시스템 목표, 범위, 성공 기준,
인수 기준, 제외 범위를 반드시 정의해야 한다.

구현 과정에서 기술적 세부사항은 구체화할 수 있지만, 관찰 가능한
동작을 바꾸려면 반드시 명세를 먼저 수정해야 한다. 코드와 명세가
충돌할 때는 명세가 명시적으로 개정될 때까지 명세를 우선한다.

### II. API 계약을 먼저 확정한다

백엔드 변경은 명시적인 API 및 데이터 계약을 기준으로 계획해야 한다.
엔드포인트, 요청 형식, 응답 형식, 검증 규칙, 상태 코드, 에러 형식,
영속성 변경, 권한 규칙은 구현 전에 반드시 문서화해야 한다.

클라이언트는 문서화되지 않은 백엔드 동작에 의존해서는 안 된다.
백엔드 코드는 관련 계약과 인수 기준을 업데이트하지 않은 채 응답,
검증, 권한 동작을 변경해서는 안 된다.

### III. 요구사항은 검증 가능해야 한다

승인된 모든 백엔드 기능은 명세와 연결된 검증 근거를 포함해야 한다.
사용자에게 보이거나 외부에서 관찰 가능한 동작은 자동화 테스트로
검증해야 하며, 자동화가 현실적으로 어렵다면 수동 검증 절차를
문서화해야 한다.

버그 수정에는 회귀 테스트를 추가해야 한다. 회귀 테스트를 추가할 수
없는 경우에는 그 이유를 명확히 설명해야 한다. 테스트는 개인 PC의
비공개 상태에 의존하지 않고 백엔드 프로젝트에서 실행 가능해야 한다.

### IV. 구현은 단순하고 유지보수 가능해야 한다

구현은 현재 명세를 만족하는 가장 작은 설계를 선택해야 한다. 다만
가까운 시점에 알려진 요구사항을 수용할 수 있는 명확한 확장 지점은
보존해야 한다.

공유 추상화는 의미 있는 중복을 줄이거나, 소유권을 명확히 하거나,
안정적인 도메인 개념을 표현할 때만 도입해야 한다. 새 의존성은
유지보수, 보안, 런타임 비용을 넘어서는 구체적 이점이 있을 때만
추가해야 한다.

### V. 도메인 무결성을 보존한다

팀 관리 동작은 컨트롤러, 데이터베이스 테이블, 전송 형식만이 아니라
명시적인 도메인 규칙을 중심으로 모델링해야 한다. 각 기능은 영향을
받는 애그리게이트 또는 도메인 개념, 허용되는 상태 전이, 호출자가
처리해야 하는 실패 상태를 반드시 식별해야 한다.

백엔드는 검증, 트랜잭션, 명확한 에러 응답을 통해 데이터 무결성을
보존해야 한다. 파괴적이거나 권한에 민감한 작업은 영향도에 맞는
보호 장치를 포함해야 한다.

### VI. 보안, 개인정보, 권한을 명시한다

팀원, 역할, 초대, 멤버십, 계정 데이터를 다루는 모든 기능은 구현 전에
접근 제어, 인증, 인가, 민감 데이터 처리 방식을 반드시 명세해야 한다.

민감한 값은 저장소에 커밋해서는 안 된다. 로그와 에러 메시지는
자격 증명, 토큰, 개인정보, 내부 전용 보안 세부사항을 노출해서는
안 된다.

### VII. 운영 가능한 동작을 만든다

기능은 개발 환경과 운영 유사 환경에서 예상 가능한 실패를 디버깅하는
데 필요한 관측 신호를 정의해야 한다. 백엔드 경계에서 반환되는 에러는
일관된 형식을 사용해야 한다.

장시간 실행 작업, 비동기 작업, 스케줄 작업, 외부 시스템에 의존하는
작업은 재시도, 타임아웃, 멱등성, 실패 동작을 반드시 명세해야 한다.

## 개발 워크플로우

1. 백엔드 기능 명세를 작성하거나 업데이트한다.
2. 인수 기준이 검증 가능해질 때까지 불명확한 부분을 정리한다.
3. 계약, 데이터 변경, 위험, 검증 전략을 포함한 구현 계획을 작성한다.
4. 독립적으로 검토할 수 있을 만큼 작은 작업 단위로 계획을 나눈다.
5. 명세, 계약, 테스트를 동기화하면서 작업을 구현한다.
6. 완료로 간주하기 전에 인수 기준을 기준으로 기능을 검증한다.

## 품질 게이트

병합 또는 릴리스 전에 모든 백엔드 기능은 다음 게이트를 통과해야 한다.

- 명세에 구현에 영향을 주는 미해결 모호성이 없어야 한다.
- API, 영속성, 권한 계약이 변경될 경우 문서화되어야 한다.
- 테스트 또는 문서화된 검증 절차가 인수 기준을 입증해야 한다.
- 보호된 데이터나 작업에는 보안 및 권한 동작이 검증되어야 한다.
- 기존 동작을 의도치 않게 회귀시키지 않아야 한다.
- 새 의존성, 마이그레이션, 운영 위험은 명시되어야 한다.

## 거버넌스

이 헌법은 `be` 프로젝트 아래의 모든 백엔드 명세, 계획, 작업,
구현을 관리한다. 자동 생성 또는 수동 작성된 모든 백엔드 SDD 산출물은
위 원칙과 품질 게이트를 통과해야 한다.

헌법을 개정하려면 이 파일을 명시적으로 수정하고, 버전을 변경하며,
기존 명세, 템플릿, 워크플로우에 미치는 영향을 요약해야 한다.

버전은 시맨틱 버저닝을 따른다.

- MAJOR: 원칙을 제거하거나 하위 호환되지 않는 방식으로 재정의한다.
- MINOR: 새 원칙을 추가하거나 거버넌스 요구사항을 실질적으로 확장한다.
- PATCH: 요구사항을 바꾸지 않고 표현만 명확히 한다.
