# Team Manager 백엔드 헌법

이 문서는 `be` 프로젝트의 SDD 기준이다. 모든 백엔드 명세, 계획,
구현, 검증은 이 원칙을 따른다.

## 핵심 원칙

### 1. 명세가 먼저다

모든 기능은 구현 전에 명세로 시작한다. 명세에는 목표, 범위, 인수
기준, 제외 범위, 모호한 지점이 포함되어야 한다. 코드가 명세와 다르면
명세를 먼저 고치고 나서 코드를 바꾼다.

### 2. API 계약을 고정한다

엔드포인트, 요청/응답 형식, 상태 코드, 에러 형식, 검증 규칙, 권한
규칙은 구현 전에 문서화한다. 프론트엔드가 추측으로 백엔드 동작에
의존하지 않게 한다.

### 3. 도메인 규칙을 드러낸다

팀, 팀원, 경기, 참가, 회비 같은 핵심 개념은 컨트롤러나 테이블보다
먼저 도메인 규칙으로 설명한다. 허용되는 상태 전이와 실패 상태를
명확히 한다.

### 4. 검증 가능해야 완료다

인수 기준은 테스트 또는 수동 검증 절차로 확인 가능해야 한다. 버그
수정에는 회귀 테스트를 추가하고, 어렵다면 이유를 남긴다.

### 5. 보안과 권한은 기본 요구사항이다

인증, 인가, 개인정보, 민감 정보, 로그 노출 여부는 기능 명세 단계에서
정의한다. 토큰, 비밀번호, 개인 정보는 저장소나 로그에 남기지 않는다.

### 6. 작고 유지보수 가능한 구현을 선택한다

현재 명세를 만족하는 가장 단순한 설계를 우선한다. 추상화와 의존성은
명확한 이점이 있을 때만 추가한다.

## 품질 게이트

- 명세에 구현을 막는 모호함이 없어야 한다.
- API, 데이터, 권한 계약이 문서화되어야 한다.
- 인수 기준을 증명하는 테스트 또는 검증 절차가 있어야 한다.
- 기존 동작을 의도치 않게 깨뜨리지 않아야 한다.
- 마이그레이션, 외부 의존성, 운영 위험을 명시해야 한다.
